Цей файл — головне джерело відповідей на питання «чому саме так». Кожен запис фіксує контекст,
прийняте рішення, розглянуті альтернативи та наслідки. Формат свідомо стислий (ADR-лайт).

## ADR-001. Модульний моноліт на NestJS

**Контекст.** Потрібен бекенд для CRUD-домену середнього розміру (авто, витрати, документи, звіти) з
RBAC і кількома інтеграціями (БД, S3). Команда мала, обсяг даних — десятки-сотні авто.

**Рішення.** Один NestJS-застосунок, розбитий на бізнес-модулі (`auth`, `users`, `vehicles`,
`vehicle-photos`, `expenses`, `documents`, `dictionaries`, `exchange-rates`, `reports`, `dashboard`,
`public`). NestJS дає DI, guards, interceptors і pipes «з коробки».

**Альтернативи.** Мікросервіси — додають мережеві межі, окремий деплой і складність без виграшу за
наявного обсягу. Безфреймворковий Express — довелось би вручну будувати DI, валідацію і структуру.

**Наслідки.** Модульність зберігає шлях до винесення окремого модуля в самостійний сервіс, якщо
навантаження зросте, але поки що все працює як один процес з однією БД.

## ADR-002. Monorepo з pnpm workspaces і спільними zod-схемами

**Контекст.** Бекенд і фронтенд мають узгоджені контракти (форми ↔ DTO). Дублювання типів і правил
валідації між двома кодовими базами швидко розсинхронізовується.

**Рішення.** Monorepo з трьох пакетів: `apps/server`, `apps/client`, `packages/shared`. У `shared`
лежать zod-схеми, типи й константи; обидва застосунки імпортують їх як `@volunteerfleet/shared`.
Одна схема валідовує і тіло запиту на сервері, і форму на клієнті.

**Альтернативи.** Два окремі репозиторії + опублікований npm-пакет контрактів — важче для одного
розробника, повільніший цикл змін. Кодген типів з OpenAPI — додатковий крок збірки; тут навпаки
OpenAPI генерується зі схем.

**Наслідки.** Один source of truth для контракту. pnpm обрано як менеджер: швидкий, економний за
дисковим простором (спільний store + hard links), нативна підтримка workspaces.

## ADR-003. ESM як єдиний стандарт модулів сервера

**Контекст.** Спочатку `apps/server` був CommonJS, а `packages/shared` — ESM. Цей мікс ламав прямий
`require()` shared із runtime сервера: скомпільований у CJS код намагався підвантажити ESM-only пакет.
Обхідні рішення (dual build shared у ESM+CJS, окремі tsconfig, post-build скрипти) лікували симптом, а
не причину — формат модулів сервера.

**Рішення.** Сервер переведено повністю на ESM: `"type": "module"`, `module/moduleResolution: NodeNext`,
`target: ES2022`. Білд робить **SWC** (через nest-cli builder) з паралельним type-check від `tsc`. Усі
відносні імпорти в TS-сорсах сервера — з `.js`-розширенням (вимога NodeNext). `packages/shared` лишається
ESM-only з conditional exports (`types → ./src/*.ts`, `import → ./dist/*.js`).

**Альтернативи.** Лишити сервер у CJS і збирати shared у двох форматах — підтримка двох версій кожного
типу/схеми, ризик розсинхрону, повільніший білд.

**Наслідки.** Чистий сучасний стандарт без обхідних рішень; сумісність із Vitest (ESM-native), AWS SDK
v3, RxJS 7, Drizzle. Ціна: `.js`-розширення в TS-імпортах; CJS-залежності (`cookie-parser`, `bcrypt`,
`helmet`) імпортуються через default; для DI-токенів у циклічних імпортах потрібен окремий tokens-файл
(інакше TDZ ламає декоратор `@Inject`).

## ADR-004. Drizzle ORM замість TypeORM/Prisma

**Контекст.** Потрібен type-safe доступ до PostgreSQL з явними міграціями. Звіти містять нетривіальні
агрегації, які краще тримати близько до SQL.

**Рішення.** Drizzle ORM з драйвером `node-postgres`. Схема описується у TypeScript-файлах
(`src/db/schema/*.ts`) і є джерелом правди; міграції генеруються через `drizzle-kit generate` (diff
схеми проти snapshot).

**Альтернативи.** _TypeORM_ — метадані на декораторах + active record погано піддаються рефакторингу.
_Prisma_ — окремий генератор і engine у runtime, додатковий крок збірки перед стартом.

**Наслідки.** Type-safe SQL builder із прямою відповідністю до SQL, прості міграції, відсутність
окремого runtime-engine. Drizzle новіший за TypeORM/Prisma — менше готових прикладів, але перевага
очевидна для типізованих SQL-важких операцій (звіти, агрегації).

## ADR-005. Drizzle-клієнт обгорнуто у `@Global()` Nest-провайдер

**Контекст.** Той самий екземпляр `Database` потрібен у багатьох сервісах; seed-скрипти, навпаки,
викликають drizzle напряму з CLI.

**Рішення.** `DbModule` (`@Global()`) створює `pg.Pool` із `DATABASE_URL` і `Database` (drizzle) над
ним, експортує DI-токени `DB_POOL` (рідкісний raw SQL) і `DB` (drizzle). `onApplicationShutdown` коректно
закриває пул.

**Альтернативи.** Окремий пул на кожен модуль — марнотратство для одного `DATABASE_URL`. Глобальний
синглтон у файлі без Nest DI — ламає дисципліну DI й не дає підкласти fake у тестах.

**Наслідки.** Будь-який сервіс інжектить `@Inject(DB) db: Database`; у unit-тестах підставляється fake
без підняття PostgreSQL.

## ADR-006. Soft delete через `deleted_at` + partial unique index

**Контекст.** Основні бізнес-сутності (`users`, `vehicles`, `expenses`, `documents`, `vehicle_photos`)
не можна видаляти фізично: на них тримаються FK з RESTRICT, а волонтерський облік потребує можливості
відновити випадково видалене.

**Рішення.** Колонки `deleted_at timestamptz` + `deleted_by uuid` (nullable). Усі SELECT за
замовчуванням додають `WHERE deleted_at IS NULL` через helper `notDeleted(table)`. Для унікальних полів —
**partial unique index** `WHERE deleted_at IS NULL` (наприклад `users.email`, `vehicles.identifier`,
`vehicles.public_slug`): дублі серед активних заборонені, але email/ідентифікатор можна перевикористати
після видалення. Адмін має `?includeDeleted=true` і `POST /:id/restore`.

**Альтернативи.** Окрема `archived_*`-таблиця — більше коду й важчі JOIN-и у звітах. Прапорець
`is_deleted boolean` — без таймстемпу втрачається «коли».

**Наслідки.** RESTRICT на FK і soft delete сумісні: логічне видалення не торкає зовнішніх ключів. Файли
у сховищі після soft delete лишаються (прибирання — поза межами першої версії).

## ADR-007. Мінімальний аудит без таблиці `audit_log`

**Контекст.** Потрібна базова відповідь на «хто і коли створив/змінив/видалив запис», але повноцінний
журнал змін із diff — окремий вертикальний зріз (механізм серіалізації, таблиця, UI перегляду).

**Рішення.** Аудит реалізовано полями `created_by` / `updated_by` / `deleted_by` (FK на `users.id`) +
таймстемпи `created_at` / `updated_at` / `deleted_at` на основних таблицях. Окремої `audit_log` немає.

**Наслідки.** Втрачається лише детальний diff («які саме поля змінилися»). Для волонтерського обсягу —
прийнятний компроміс. Повернути `audit_log` з `payload_diff` і UI «Журнал дій» можна пізніше без зміни
наявної моделі.

## ADR-008. `updated_at` через Drizzle `$onUpdate`, а не тригери

**Контекст.** Конвенція — підтримувати `updated_at` на кожній таблиці.

**Рішення.** У схемі `.$onUpdate(() => new Date())` на колонці `updated_at`: JS-функція виконується
перед кожним `UPDATE` через Drizzle.

**Альтернативи.** Postgres TRIGGER + FUNCTION — гарантує оновлення навіть при ручному `psql UPDATE`, але
важче переносити між середовищами. Ручне `set({ updatedAt })` у сервісах — boilerplate, легко забути.

**Наслідки.** Прямі `UPDATE` поза Drizzle не оновлять `updated_at`. Прийнятно: усі мутації йдуть через
сервіси.

## ADR-009. Мультивалюта зі збереженим курсом на кожній витраті

**Контекст.** Волонтерські витрати часто йдуть у USD/EUR (закупівлі за кордоном), а звіти треба зводити
у гривні. Курс змінюється з часом, тому централізована таблиця курсів зробила б історичні звіти
нестабільними при оновленні даних.

**Рішення.** Підтримуються UAH (базова) + USD + EUR. На `expenses` додано `rate numeric(14,6)` (курс до
UAH на момент події) і `rate_source enum('default','manual')`. Курси беруться з локального JSON-файлу
`apps/server/data/exchange-rates.json` з помісячною деталізацією. Звіти зводяться у UAH через
`amount × rate`. Деталі — у [currency.md](currency.md).

**Альтернативи.** Таблиця `exchange_rates` у БД — складніше супроводжувати; для першої версії JSON
дешевший. Інтеграція з API НБУ — інфраструктура на потім. Денний курс — для волонтерських обсягів
помісячного достатньо.

**Наслідки.** Збережений на запис курс гарантує, що вже сформований звіт не зміниться після оновлення
JSON. Список валют розширюється додаванням елемента в масив + рядків у JSON + опції в UI.

## ADR-010. Історичний курс не перераховується автоматично при редагуванні витрати

**Контекст.** Користувач може змінити дату чи валюту витрати після створення. Автоматичний перерахунок
курсу виглядав би зручним, але руйнував би історичність фінансового запису.

**Рішення.** `POST /expenses` підставляє дефолтний курс із `ExchangeRatesService`, якщо `rate` не
переданий явно (інакше `rate_source = 'manual'`); для UAH завжди `rate = 1`. `PATCH /expenses/:id` **не**
тягне новий курс автоматично — UI має явно передати новий `rate`, якщо користувач натиснув «скинути до
дефолтного».

**Наслідки.** Збережений курс лишається історичним фактом; звіти за минулі періоди стабільні.

## ADR-011. Бізнес-семантика статусу через `vehicle_status_kind`

**Контекст.** Дашборд рахує «авто в роботі» і «передано». Прив'язувати ці агрегації до тексту назви
статусу (`"в ремонті"`, `"передано"`) крихко: назву задає адмін у довіднику й може її змінити.

**Рішення.** До `vehicle_statuses` додано enum-поле `kind`: `in_work` (проміжні робочі стани), `final`
(кінцеві стани життєвого циклу), `other` (дефолт для некласифікованих). Дашборд агрегує за `kind`, а не
за назвою. Додано також `color` як презентаційне поле для тегів/прогрес-барів.

**Альтернативи.** Хардкод відомих назв статусів — ламається при перейменуванні чи зміні мови UI.

**Наслідки.** Адмін вільно перейменовує статуси без ризику зламати аналітику. `final` навмисно
нейтральний: покриває не лише «Передано», а й можливі майбутні стани («Списано», «Втрачено»). Жорсткого
обмеження «рівно один final» немає — дашборд підсумовує 0..N фінальних статусів.

## ADR-012. Фото авто — окрема таблиця `vehicle_photos`, а не документи

**Контекст.** Фото авто не є підтверджувальними документами: вони показуються галереєю (зокрема на
публічній сторінці), мають ручний порядок і ліміт до 10 на авто. Змішування з `documents` ускладнило б
фільтри й публічний контур.

**Рішення.** Окрема таблиця `vehicle_photos` (`vehicle_id`, `file_key`, `mime_type`, `size_bytes`,
`sort_order` + audit/soft-delete поля). Файли — у тому самому MinIO через `StorageService`, з
key-префіксом `vehicle-photos/{vehicleId}/...`. Приймаються лише image-MIME з whitelist; ліміт 10
активних фото перевіряється у транзакції з блокуванням рядка авто.

**Альтернативи.** Зберігати фото в `documents` з прапорцем типу — заплутує фільтри й публічну логіку.

**Наслідки.** Наявні документи-зображення не мігрують у `vehicle_photos`; за потреби — окрема операція
post-MVP.

## ADR-013. Призначення документа виводиться з `expense_id`, а не зберігається окремим полем

**Контекст.** У картці авто треба відрізняти загальні документи від документів конкретної витрати. Зв'язок
`documents.expense_id` уже існує, тому окремий enum/прапорець призначення дублював би стан.

**Рішення.** Документ вважається документом витрати, якщо `expense_id IS NOT NULL`; загальним — якщо
`expense_id IS NULL` і є прив'язка до авто. Фільтр «Загальні / Витрати» реалізовано поверх уже
завантаженого списку документів авто; backend має фільтр за `expenseId`.

**Наслідки.** Немає ризику розсинхрону між «призначенням» і фактичним FK. Якщо з'являться інші
призначення (акт передачі, сервісна історія) — тоді знадобиться окрема класифікація.

## ADR-014. JWT: access у пам'яті + refresh у httpOnly cookie з ротацією

**Контекст.** Stateless-автентифікація для SPA. Питання — де зберігати токени, як ротувати, який
мінімальний рівень безпеки прийнятний.

**Рішення.**

- **Access JWT** (HS256, TTL 15 хв) повертається в тілі `POST /auth/login` і зберігається в пам'яті
  фронта (Zustand store), **не** в localStorage.
- **Refresh JWT** (HS256, TTL 30 днів) віддається лише як `Set-Cookie: refresh_token`
  (`HttpOnly; Secure` у prod; `SameSite=Strict; Path=/api/v1/auth`).
- `POST /auth/refresh` **ротує** refresh: кожен виклик видає новий токен і ставить нову cookie. У payload
  є `jti` (uuid) — гарантує унікальність токена й закладає основу для майбутньої денілист-логіки.
- Після перезавантаження сторінки фронт робить тихий `POST /auth/refresh` (cookie приходить автоматично).

**Альтернативи.** Refresh у localStorage — вразливий до XSS. Серверна сесія (Redis) — більше
інфраструктури і відхід від stateless JWT.

**Наслідки.** localStorage не використовується для токенів; `SameSite=Strict` захищає `/auth/refresh` від
CSRF. Денілист для миттєвого logout усіх пристроїв — поза межами першої версії (ротація робить старий
refresh невалідним при наступному використанні).

## ADR-015. Access-токен дублюється в httpOnly cookie + Vite-проксі (для нативних завантажень браузера)

**Контекст.** За [ADR-014](#adr-014-jwt-access-у-памяті--refresh-у-httponly-cookie-з-ротацією) access жив
лише в пам'яті й додавався axios-інтерсептором як `Authorization: Bearer`. Це ламається для запитів, які
робить **сам браузер**, а не axios: завантаження документа (`window.location`), прев'ю (`<img src>`),
`window.open`. У такі запити заголовок не підкласти → guard повертає `401`. Додатково: клієнт (`:5173`) і
сервер (`:3000`) — різні origin, тож cross-site cookie потребувала б `SameSite=None`, що знімає захист.

**Рішення.**

- Access-токен **додатково** кладеться в httpOnly cookie `access_token` (`Path=/api/v1`,
  `SameSite=Strict`, `Secure` у prod) на login/refresh; чиститься на logout.
- `JwtAuthGuard` читає токен із `Authorization: Bearer`, а як fallback — із cookie `access_token`. Bearer
  лишається основним для XHR/axios; cookie покриває нативні завантаження.
- **Vite dev-proxy** `/api → http://localhost:3000`: клієнт ходить на відносний `/api/v1`, усе стає
  same-origin, cookie — first-party (`SameSite=Strict` працює без HTTPS у dev). `VITE_API_URL=/api/v1`.

**Наслідки.** Прев'ю й завантаження працюють через нативні браузерні запити; токен у httpOnly cookie
недосяжний для XSS-exfiltration. Ціна: авто-надсилання cookie відкриває CSRF-вектор на state-changing
запити — для першої версії мітигується `SameSite=Strict` + same-origin proxy; повноцінний CSRF-захист
(token / double-submit) — свідомий борг на майбутнє. У prod cookie вимагає HTTPS і розгортання FE+BE за
одним доменом / reverse-proxy.

## ADR-016. Публічний read-only контур як окремі санітизовані DTO

**Контекст.** Потрібен анонімний перегляд частини даних (для донорів) без доступу до back-office CRUD.
Внутрішні DTO авто, витрат і документів містять приватні поля: VIN, службовий опис, `created_by`,
конкретні витрати, документи.

**Рішення.** Окремий `/public/*` контур із `@Public()` ендпоінтами й власними zod-схемами
(`publicVehicleResponseSchema`, `publicFundingReportResponseSchema`). Публічна картка авто повертається
лише для `is_public = true`, `public_slug = :slug`, `deleted_at IS NULL`. Публічний звіт по джерелу
віддає тільки агрегати (`totalUah`, `byCategory`, `byVehicle`) і тільки публічні авто; список конкретних
витрат і документи не віддаються. Якщо умови не виконані — `404` (щоб не розкривати існування запису).

**Альтернативи.** Розширювати (`extend`) внутрішні схеми й «обрізати» поля — ризик випадково відкрити нове
приватне поле в майбутньому.

**Наслідки.** Невелике дублювання мапінгу в `PublicModule`, зате межа доступу очевидна в коді й
тестується окремо. Це приклад data minimization: публічний контракт будується від потреб гостя, а не
обрізанням внутрішнього.

## ADR-017. Файли — через S3-сумісне сховище (MinIO) з presigned-завантаженням

**Контекст.** Документи можуть бути завантаженим файлом або зовнішнім посиланням. Локально потрібне
сховище без прив'язки до конкретного провайдера в production.

**Рішення.** `StorageModule` інжектить AWS SDK `S3Client` з `forcePathStyle: true` (для MinIO).
`DocumentsService` підтримує два режими: `upload` (файл за `file_key` у bucket) і `link` (зовнішній
`url`). Завантаження не стрімиться через backend: ендпоінт повертає `302` на presigned-URL (короткоживучий)
або на зовнішнє посилання. Ключ файлу — `documents/{document_uuid}/{sanitized_name}`; UUID генерується до
upload, щоб ключ і запис у БД мали спільний ідентифікатор. MIME перевіряється content-sniffing
(`file-type`) поверх whitelist — `Content-Type` клієнта не є джерелом довіри.

**Альтернативи.** Зберігати файли в БД (bytea) — роздуває базу, ускладнює бекапи. Стрімити завантаження
через backend — робить його bottleneck.

**Наслідки.** Той самий код працює з AWS S3 / Cloudflare R2 / Backblaze B2 у проді без зміни доменної
логіки. При soft delete файл фізично лишається в bucket; cleanup job — поза межами першої версії.

## ADR-018. Звіти через print-stylesheet, без PDF-бібліотеки

**Контекст.** Потрібен експорт звітів у форматі, придатному для друку й передачі донорам, з коректною
кирилицею.

**Рішення.** Звіт — окрема React-сторінка з `@media print` CSS і кнопкою «Зберегти як PDF», яка викликає
`window.print()`. Користувач зберігає як PDF через нативний діалог браузера.

**Альтернативи.** _puppeteer_ — headless Chromium у образі (важко). _pdfmake/pdfkit_ — програмна верстка
+ проблеми з кириличними шрифтами. _exceljs_ — лише XLSX.

**Наслідки.** Жодного headless Chromium і PDF-залежностей; кирилиця «з коробки»; стилі Ant Design
перевикористовуються без дублювання. За потреби XLSX-експорт можна додати окремо пізніше.

## ADR-019. Спільна zod-валідація на бекенді й фронтенді

**Контекст.** Правила валідації (довжини, формати, обов'язковість) мають бути однакові на сервері й у
формах.

**Рішення.** zod-схеми у `packages/shared` — джерело правди. На сервері `nestjs-zod`
(`ZodValidationPipe`) валідує тіло/query/params і генерує OpenAPI. На клієнті та сама схема валідує форму
Ant Design через адаптер `zodValidator`, а помилки бекенда у форматі `details: [{ path, message }]`
мапляться у `form.setFields` так само, як локальні zod-issues.

**Наслідки.** Немає дублювання валідації; контракт API типобезпечний на обох кінцях. OpenAPI/Swagger
генерується автоматично і не розходиться зі схемами.

## ADR-020. Гібридний запуск: застосунки локально, інфраструктура в Docker

**Контекст.** Потрібен швидкий цикл «зміна коду → ефект» для розробки і водночас відтворюване сховище
(PostgreSQL, MinIO), однакове між розробниками й CI.

**Рішення.** `pnpm dev` запускає сервер і клієнт локально (watch/HMR); `docker compose up` піднімає
PostgreSQL + MinIO (+ опційно pgAdmin). Самі застосунки не докеризуються для розробки.

**Альтернативи.** Усе в Docker — повільніший HMR, більше образів писати й підтримувати. Усе локально
(вкл. БД) — розходження середовищ між розробниками.

**Наслідки.** Менше Docker-образів; інфраструктура ідентична між учасниками. Production-схема (reverse
proxy + TLS) описана в [deployment.md](deployment.md) як цільовий сценарій.

## ADR-021. Розподіл стану на клієнті: TanStack Query (server state) + Zustand (UI state)

**Контекст.** Списки авто/витрат/документів/звітів — це серверний стан із кешуванням та інвалідацією.
Глобального UI-стану мало (auth-користувач на клієнті, відкриті модалки, активні фільтри).

**Рішення.** **TanStack Query** для всього серверного стану (кеш, інвалідація після мутацій, retry,
фонове оновлення). **Zustand** для невеликого глобального UI-стану — ~1 KB, без провайдерів і
boilerplate.

**Альтернативи.** Redux / RTK — забагато boilerplate за наявного обсягу глобального стану. Тримати
серверні дані в Redux — дублювати те, що RQ робить краще.

**Наслідки.** Auth-store не персиститься в localStorage (із міркувань безпеки, [ADR-014](#adr-014-jwt-access-у-памяті--refresh-у-httponly-cookie-з-ротацією));
після перезавантаження — тихий refresh.

## Кандидати на майбутні рішення

Нетривіальні моменти, які варто оформити окремими ADR при розвитку проєкту:

- CSRF-захист (token / double-submit) для cookie-auth і денілист refresh-токенів (борг із [ADR-015](#adr-015-access-токен-дублюється-в-httponly-cookie--vite-проксі-для-нативних-завантажень-браузера) / [ADR-014](#adr-014-jwt-access-у-памяті--refresh-у-httponly-cookie-з-ротацією)).
- Повернення детального `audit_log` з diff і UI «Журнал дій» ([ADR-007](#adr-007-мінімальний-аудит-без-таблиці-audit_log)).
- Інтеграція із зовнішнім API курсів валют замість/на додачу до JSON ([ADR-009](#adr-009-мультивалюта-зі-збереженим-курсом-на-кожній-витраті)).
- Cleanup-job для прибирання осиротілих файлів у сховищі після soft delete ([ADR-017](#adr-017-файли--через-s3-сумісне-сховище-minio-з-presigned-завантаженням)).
- Структуровані JSON-логи (`request_id` correlation) і зовнішній моніторинг.
