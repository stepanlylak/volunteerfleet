# Епік: Групи документів (Document Groups)

> **Статус:** Proposed (готовий до декомпозиції в спринти)
> **Тип:** Епік / архітектурна зміна
> **Зачіпає:** `apps/server`, `apps/client`, `packages/shared`, схему БД
> **Трекінг:** планування ведеться в репозиторії (`docs/epics/`); зовнішній трекер не використовуємо.
> **Мова документа:** українська (наратив), англійська (ідентифікатори, таблиці, колонки, enum'и) — за правилами проєкту.
> **Повʼязані епіки:** [`vehicle-status-flow`](./vehicle-status-flow.md) (флоу зміни статусу й слоти документів), [`vehicle-galleries`](./vehicle-galleries.md) (аналог 1:N gallery → items).

Цей документ — єдине джерело правди для епіку. Усі рішення тут уже узгоджені з продакт-овнером;
розділ «Поза обсягом» так само обовʼязковий, як і обсяг.

---

## 1. Контекст і проблема

При зміні статусу авто ми кріпимо підтверджувальні документи (техпаспорт, акти приймання-передачі,
митні декларації тощо). Зараз кожен такий документ — це **окрема FK-колонка «один слот = рівно один
документ»** на `vehicle_status_history`: `registrationDocId`, `stampedRegistrationDocId`,
`customsDeclarationDocId`, `stampedCustomsDeclarationDocId`, `transferActDraftDocId`,
`transferActSignedDocId`, `returnActDocId`
([`vehicle-status-history.ts`](../../apps/server/src/db/schema/vehicle-status-history.ts)).

Сама таблиця `documents` уже вміє те, що треба для одного запису
([`documents.ts`](../../apps/server/src/db/schema/documents.ts)):

- `kind = 'upload'` (є `file_key`) **або** `kind = 'link'` (є `url`) — файл і посилання взаємозамінні;
- `document_type` (`registration_certificate`, `transfer_act_signed`, …).

**Проблема — кардинальність.** Один логічний документ ≠ один файл. Техпаспорт — це може бути 1 PDF,
1 посилання, або 2–3 фото. Так само акти. Поточна модель «слот = один документ» цього не виражає.

Бракує саме рівня **1:N: логічний документ (група) → N фізичних артефактів**, де кожен артефакт
незалежно є upload або link.

## 2. Мета і нефункціональні вимоги

**Мета.** Дати можливість кріпити до одного логічного слота **довільну кількість** файлів/посилань,
обʼєднаних у групу. Зробити «групу документів» **перевикористовною** концепцією (статуси, витрати,
загальні документи машини), а не точковою фічею під статуси.

**Нефункціональні вимоги:**

- **Тенант-ізоляція** — група завжди в межах організації (`organization_id`) і завжди привʼязана до
  конкретного авто (`vehicle_id`).
- **Узгодженість із наявними патернами** — 1:N моделюємо так само, як `vehicle_galleries` → `vehicle_gallery_items`
  (дитина тримає FK на батька).
- **Перевикористання `documents`** — групуємо записи `documents` (бо вони підтримують link), а **не**
  `vehicle_gallery_items` (там лише `file_key`, без посилань).

## 3. Зафіксовані рішення (конституція епіку)

Ці рішення вже прийняті спільно з продакт-овнером. Не переглядати під час імплементації без окремого
узгодження.

1. **Модель — окрема таблиця груп + FK з документа на групу (1:N).**
   `documents.group_id → document_groups.id`. Документ належить **максимум одній** групі; група фанаутиться
   на N документів. **Без** таблиці-прокладки (M:N не потрібен) і **без** напрямку «група → один документ».
2. **Слоти статусу вказують на групу, а не на документ.** Колонки `*DocId` у `vehicle_status_history`
   стають `*GroupId` (FK на `document_groups`, `onDelete: restrict`). Ідентичність слота = «яка колонка
   вказує на групу» (це заодно знімає теперішню двозначність, де `registration_certificate` як `document_type`
   використовується двічі — на `paid` і на `arrived`).
3. **Скоуп групи.** `document_groups.vehicle_id` — **NOT NULL** (група завжди належить авто),
   `organization_id` — NOT NULL. Контекст витрати зберігається на `expenses.document_group_id`;
   одна група може бути перевикористана кількома витратами.
4. **«Завжди група» для слотів.** Слот завжди → група; група з одним PDF усередині — норма. Один шлях у коді.
   Тому контексти «статус» і «витрати» групують **примусово** (див. UI, розд. 6).
5. **Перевикористання існуючих файлів = перенос (move).** Якщо при завантаженні з увімкненим групуванням
   юзер обирає **вже згрупований** існуючий документ — його `group_id` **переписується** на нову групу
   (документ вилітає зі старої). Це покриває кейс «залив чеки наперед → потім створив витрату й підтягнув їх».
   M:N (документ у кількох групах одночасно) — **поза обсягом**.
6. **Спустошена група видаляється повністю (hard `DELETE`).** Якщо після переносу в старій групі лишилось
   0 активних документів — старий запис групи видаляється **назавжди** (не soft-delete): сам файл лишається
   в новій групі, а порожній запис групи тримати немає сенсу. Це безпечно, бо guard (рішення 7) гарантує,
   що в такої групи немає вхідних FK.
7. **Guard: статусні докази недоторкані.** Документ, чия група привʼязана до слота
   `vehicle_status_history.*GroupId`, **не можна** забрати переносом — інакше транзиція статусу лишиться
   без обовʼязкового документа. Реалізація: пікер «обрати існуючі» **не показує** такі документи; на бекенді —
   відмова 400, якщо запит усе ж дійшов.

## 4. Модель даних

### 4.1 Нова таблиця `document_groups`

```
document_groups (
  id               uuid pk
  organization_id  uuid not null  → organizations.id (restrict)
  vehicle_id       uuid not null  → vehicles.id      (restrict)   -- рішення 3
  name             varchar(255) null                              -- опційна мітка ("Техпаспорт")
  created_by       uuid not null  → users.id
  updated_by       uuid not null  → users.id
  created_at / updated_at  timestamptz
  -- БЕЗ deleted_at: групи не soft-delete'имо (рішення 6); порожні видаляються hard-delete
)
indexes:
  document_groups_org_vehicle_idx (organization_id, vehicle_id)
```

> Примітка: на відміну від `vehicle_galleries`/`documents`, у `document_groups` **немає** `deleted_at`.
> Група існує, лише поки в ній є документи (для статусних — поки існує слот). Порожня група — артефакт без
> сенсу, тож видаляється фізично.

### 4.2 Зміни в `documents`

```
+ group_id  uuid null → document_groups.id (restrict)
```

Розширити чек `documents_attached_to_something`:

```
vehicle_id IS NOT NULL OR group_id IS NOT NULL
```

Колонка `documents.expense_id` видаляється. Документ витрати знає контекст через групу.

### 4.3 Зміни в `expenses`

```
+ document_group_id uuid null → document_groups.id (restrict)
```

Одна витрата має максимум одну групу документів. Кілька витрат можуть посилатися на ту саму групу.
`document_groups.vehicle_id` має збігатися з `expenses.vehicle_id`.

### 4.4 Зміни в `vehicle_status_history`

Перейменувати 7 колонок-слотів `*DocId → *GroupId`, FK на `document_groups`, `onDelete: restrict`:

| Було (`→ documents.id`)              | Стане (`→ document_groups.id`)         |
| ------------------------------------ | -------------------------------------- |
| `registration_doc_id`                | `registration_group_id`                |
| `stamped_registration_doc_id`        | `stamped_registration_group_id`        |
| `customs_declaration_doc_id`         | `customs_declaration_group_id`         |
| `stamped_customs_declaration_doc_id` | `stamped_customs_declaration_group_id` |
| `transfer_act_draft_doc_id`          | `transfer_act_draft_group_id`          |
| `transfer_act_signed_doc_id`         | `transfer_act_signed_group_id`         |
| `return_act_doc_id`                  | `return_act_group_id`                  |

### 4.5 Звʼязки (підсумок)

```
vehicle_status_history.<slot>GroupId ──▶ document_groups ◀── documents.group_id
                       (1:1, на групу)                       (N:1, діти групи)
                                              │
                                    vehicle_id (NOT NULL)
                                              ▲
                              expenses.document_group_id (N:1)
```

## 5. Бекенд

### 5.1 Валідація транзиції статусу

У [`vehicle-transition.service.ts`](../../apps/server/src/modules/vehicles/vehicle-transition.service.ts)
перевірка `docChecks` (зараз «документ існує, має тип, належить авто») переїжджає на рівень групи:

- група належить цьому `vehicle_id` та `organization_id`;
- група містить **≥ 1** активний документ (`deleted_at IS NULL`);
- (опційно) верхня межа на к-сть айтемів у групі.

Те саме дзеркалити в `editStatusHistory`.

### 5.2 Правило «не переюзати акт передачі»

Зараз: `returned → transferred` забороняє той самий `transferActSignedDocId`, що й попередня передача.
Стає: забороняє ту саму **групу** (`transfer_act_signed_group_id`), що й попередня передача.

### 5.3 Сервіс переносу документа (move)

Одна транзакція:

```
1. джерельна група належить слоту vehicle_status_history.*GroupId?  → 400 (рішення 7)
2. document.group_id := <нова група>
3. count(active docs у старій групі) == 0 ?  → DELETE старої групи (рішення 6)
```

Пікер «обрати існуючі файли» на рівні запиту фільтрує документи: лише ті, чия група **не** зайнята слотом
історії (плюс, зрозуміло, у межах того ж авто/організації).

## 6. Frontend: компонент-аплоадер і чекбокс «групувати»

Аплоадер — велика форма: нові завантаження + вибір існуючих файлів + URL. Поведінка групування керується
**пропом**, щоб у різних контекстах різнилась.

### 6.1 Проп `grouping`

```ts
type GroupingProp =
  | { mode: 'locked'; grouped: boolean } // чекбокс відмальований, disabled
  | { mode: 'optional'; default: boolean } // чекбокс активний, default — початковий стан
  | { mode: 'hidden' }; // без чекбокса (на майбутнє)
```

### 6.2 Поведінка по контекстах

| Контекст         | Проп                                  | Дефолт     | Чекбокс             |
| ---------------- | ------------------------------------- | ---------- | ------------------- |
| Зміна статусу    | `{ mode: 'locked', grouped: true }`   | згруповано | показаний, disabled |
| Витрати          | `{ mode: 'locked', grouped: true }`   | згруповано | показаний, disabled |
| Документи машини | `{ mode: 'optional', default: true }` | згруповано | активний            |

### 6.3 Семантика чекбокса (на рівні даних)

- **ON** → весь батч (нові аплоади + вибрані існуючі + URL) → в **одну** групу.
- **OFF** → кожен айтем — самостійний документ (`group_id = null`).

(Тому статус/витрати — `locked: true`: слот/витрата вказують на групу, тож група має існувати завжди,
навіть для одного файлу.)

### 6.4 Конфірм переносу

`Modal.confirm` показуємо **лише** коли вибраний існуючий файл уже належить групі. Текст адаптивний:

- стара група спустіє: _«Файл „X" зараз у групі „Y". Перенести сюди? Група „Y" залишиться порожньою і буде видалена.»_
- у старій групі лишаться інші файли: _«Файл „X" буде перенесено з групи „Y" сюди.»_

## 7. shared (Zod / DTO)

- Контракт `DocumentGroup` (id, organizationId, vehicleId, expenseIds[], name?, documents[]).
- `ExpenseResponse.documentGroupId` повертає групу документів витрати.
- Upload/link DTO не містять `expenseId`; контекст витрати задається через групу.
- DTO транзиції: поля `*GroupId` замість `*DocId`; решта схеми транзиції без змін.
- DTO створення/оновлення групи + операції переносу документа.

## 8. Міграція

Живих продакшн-даних немає (узгоджено в epic `organizations-multitenancy`), але dev/test-сценарій:

1. Створити `document_groups`, додати `documents.group_id`, розширити чек.
2. Перейменувати `*DocId → *GroupId` у `vehicle_status_history`; для наявних значень — обгорнути кожен
   документ у **групу-одинак** (одна група на один документ) і перепосилатись на неї.
3. Перестворити dev/test БД за потреби (без бекфілу).

Для переходу документів витрат:

1. Додати `expenses.document_group_id`.
2. Для кожної витрати з прямими документами створити або перевикористати одну групу, перенести документи
   в `documents.group_id`, проставити `expenses.document_group_id`.
3. Видалити `document_groups.expense_id` і `documents.expense_id`.

SQL хендтюнимо на ідемпотентність (за нотаткою про drizzle-workflow).

## 9. Поза обсягом

- **M:N** (один документ у кількох групах одночасно) — навмисно ні; перенос (move) покриває реальні кейси.
- **Злиття з `vehicle_galleries`** — галереї лишаються окремо (публічні фото, обкладинки, підписи — інша задача).
- **Per-org довідники типів документів**, кастомні слоти — поза обсягом.
- **«Обкладинка» групи** (аналог `coverItemId`) — не вводимо, поки немає потреби.
