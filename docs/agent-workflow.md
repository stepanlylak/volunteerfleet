# Воркфлоу роботи з тікетами через AI-агентів

Загальний процес для **всіх** тікетів, які виконуються AI-агентами в цьому репозиторії — незалежно від
епіку. Епіки декомпозуються на тікети-`*.md` (вони грають роль issue; зовнішнього трекера не
використовуємо), а цей файл описує, як ці тікети рухаються від створення до merge.

> Загальні git/commit/PR-конвенції — у [contributing.md](contributing.md). Тут лише специфіка
> агентного процесу: статуси, хто що робить, як розблоковуються залежні тікети.

## Ролі

- **Агент** — бере один тікет, реалізує його в окремій гілці (або worktree), комітить і пушить.
- **Оркестратор** — координує тікети, рев'юить роботу агента, **створює PR і мерджить** його в `main`.
  Агент PR **не** відкриває.

## Статуси тікета

Поле `status` у frontmatter кожного тікета:

| Статус    | Значення                                                                                             |
| --------- | ---------------------------------------------------------------------------------------------------- |
| `todo`    | тікет створено, але він ще **не готовий до девелопменту**: не всі залежності (`depends_on`) виконані |
| `ready`   | усі залежності виконані — тікет можна брати в роботу                                                 |
| `done`    | агент вважає тікет виконаним: закомітив і запушив гілку. PR створює й мерджить оркестратор           |
| `blocked` | заблоковано з іншої причини (вкажіть її в тілі тікета)                                               |

Окремого `in-progress` немає: робота завжди йде в окремій гілці/worktree, тож «у роботі» не трекається
статусом. Окремого `in-review` теж немає — рев'ю відбувається над PR, який створює оркестратор.

### Як тікети стають `ready`

Тікет переходить `todo → ready`, коли **всі** його `depends_on` мають статус `done`. Відповідальність за
це лежить на агенті, що завершує тікет: закриваючи свій тікет, він **переводить у `ready` всі тікети, які
цим розблокував** (тобто ті, у кого це був останній незавершений `depends_on`). Так після merge його PR
інші агенти одразу бачать, що можна брати наступні тікети.

## Життєвий цикл тікета

1. **Взяти** тікет зі статусом `ready`.
2. `git fetch origin` → відгалузити від `main` гілку `<type>/<id>-<slug>`
   (напр. `feat/org-5-jwt-active-org`); `type` — з frontmatter тікета. За потреби — окремий worktree.
   Заповнити `branch:` у тікеті.
3. Реалізувати за розділом **Обсяг** тікета; виконати **Критерії приймання** + PR-чекліст із
   [contributing.md](contributing.md) (`pnpm lint`, `typecheck`, `test`; міграції; docs).
4. Перевести в `ready` всі тікети, які цей розблокував (див. вище).
5. Коли агент вважає тікет виконаним: виставити `status: done`, **закомітити і запушити** гілку.
   PR **не** відкривати.
6. Оркестратор рев'юить, **створює PR і мерджить**, якщо все гаразд, і заповнює `pr:`. Якщо потрібні
   правки — повертає тікет у `ready` з коментарем, агент доопрацьовує в тій самій гілці.

> `done` ≠ «прийнято»: це сигнал «робота агента завершена, гілка запушена». Прийняти зміни (merge) або
> повернути на доопрацювання вирішує оркестратор.

## Поля frontmatter тікета

- `status` — див. таблицю вище.
- `depends_on` — список id тікетів, від яких залежить цей. Поки не всі вони `done`, статус — `todo`.
- `branch` — заповнює агент, коли бере тікет у роботу.
- `pr` — заповнює оркестратор після створення PR.
