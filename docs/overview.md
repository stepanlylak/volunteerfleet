# Огляд системи

## Що це

**VolunteerFleet** — вебплатформа для обліку транспортних засобів, переданих на потреби Збройних
Сил України волонтерськими ініціативами. Це єдиний реєстр авто, витрат на них, підтверджувальних
документів і звітів для донорів.

Типова проблема, яку закриває система: волонтерська ініціатива знаходить, купує, ремонтує та передає
десятки авто, фінансованих із різних зборів. Облік ведеться у таблицях і чатах, гроші проходять у
кількох валютах, а донори періодично просять прозвітувати «куди пішли кошти». VolunteerFleet зводить
це в одну систему: кожне авто має життєвий цикл зі статусами, кожна витрата прив'язана до авто,
категорії, джерела фінансування і збереженого курсу валют, а звіт по авто чи джерелу формується за
кілька кліків.

## Ролі та сценарії

| Роль          | Доступ                                                                                                           |
| ------------- | --------------------------------------------------------------------------------------------------------------- |
| **admin**     | Повний доступ: керування користувачами і довідниками, soft delete / restore, скидання паролів.                  |
| **volunteer** | Створення й редагування авто, витрат, документів і фото; перегляд звітів.                                        |
| **guest**     | Read-only доступ до реєстру авто (список, картка, історія статусів) у back-office.                               |
| _анонім_      | Публічні read-only сторінки (`/public/...`) для авто, які адміністратор явно позначив публічними — без логіну.   |

Основні сценарії:

- **Облік авто.** Завести авто (марка, модель, рік, VIN, внутрішній ідентифікатор), вести його статус
  (знайдено → куплено → в ремонті → готове → передано) з історією переходів, додавати фото.
- **Облік витрат.** Зафіксувати витрату у фактичній валюті (UAH/USD/EUR) із збереженим курсом до UAH,
  прив'язати до авто, категорії та джерела фінансування, прикріпити чек/акт.
- **Документи.** Завантажити файл у сховище або додати зовнішнє посилання; прив'язати до авто та/або витрати.
- **Звітність.** Сформувати звіт по авто (зведення витрат, історія, документи) або по джерелу фінансування
  за період; роздрукувати / зберегти як PDF.
- **Публічне інформування.** Показати донорам публічну сторінку авто з прогресом збору — без розкриття
  внутрішніх деталей (VIN, конкретні витрати, службові поля).

## Архітектурний стиль

Система — **модульний моноліт** на бекенді + **SPA** на фронтенді, які поділяють спільні контракти.

```
┌──────────────────────────┐         ┌───────────────────────────────────────────┐
│  Браузер (SPA, React)    │  HTTPS  │  Backend (NestJS, модульний моноліт)        │
│                          │ ──────▶ │                                             │
│  Ant Design UI           │  REST   │  auth · users · vehicles · vehicle-photos   │
│  TanStack Query (cache)  │ /api/v1 │  expenses · documents · dictionaries        │
│  Zustand (UI state)      │ ◀────── │  exchange-rates · reports · dashboard       │
│  axios (+ refresh on 401)│  JSON   │  public · health · storage                  │
└──────────────────────────┘         └───────────────┬───────────────┬─────────────┘
            ▲                                         │               │
            │ shared zod-схеми (контракт)             │ Drizzle       │ @aws-sdk/client-s3
            │                                         ▼               ▼
   ┌────────┴─────────┐                       ┌──────────────┐  ┌──────────────┐
   │ packages/shared  │                       │ PostgreSQL 16│  │ MinIO (S3)   │
   │ zod · типи · конст│                       └──────────────┘  └──────────────┘
   └──────────────────┘
```

- **Backend — NestJS.** Бізнес-логіка розкладена по модулях (DI, guards, interceptors, pipes). Моноліт,
  а не мікросервіси: обсяг домену не виправдовує мережевих меж між сервісами, але модульність зберігає
  можливість винести модуль у окремий сервіс пізніше.
- **Frontend — React SPA.** Окремий клієнт на Vite; уся взаємодія з сервером — через REST `/api/v1`.
- **packages/shared.** zod-схеми, типи й константи, що імпортуються і бекендом, і фронтендом. Це єдине
  джерело правди для контракту API: одна схема валідовує і вхідні дані на сервері, і форму на клієнті.
- **PostgreSQL** — основне сховище даних; **MinIO** (S3-сумісне) — сховище файлів.

Деталі та обґрунтування кожного з цих виборів — у [architecture-decisions.md](architecture-decisions.md).

## Потік даних: приклад «створення витрати»

1. Користувач відкриває форму витрати, обирає дату й валюту. Клієнт робить
   `GET /api/v1/exchange-rates?date=…&currency=USD` і автозаповнює поле курсу.
2. Форма валідовується спільною zod-схемою (`expenseCreateSchema` з `packages/shared`).
3. `POST /api/v1/expenses` → `ZodValidationPipe` валідує тіло тією ж схемою; `JwtAuthGuard` і
   `RolesGuard` перевіряють доступ.
4. `ExpensesService` підставляє курс (`rate`) і його джерело (`rate_source`), проставляє `created_by`
   з контексту користувача і пише запис через Drizzle.
5. У відповідь — повна репрезентація витрати; TanStack Query на клієнті інвалідовує кеш списків.

## Карта домену (сутності)

- **User** — користувач системи (`admin` / `volunteer` / `guest`).
- **Vehicle** — авто; має статус, історію статусів, фото, витрати й документи; може мати публічний контур.
- **VehicleStatus** (довідник) — статус авто з бізнес-семантикою (`kind`) і кольором.
- **VehicleStatusHistory** — журнал переходів статусу авто.
- **VehiclePhoto** — фото авто (галерея з порядком), окремо від документів.
- **Expense** — витрата з валютою, збереженим курсом, категорією та джерелом фінансування.
- **ExpenseCategory**, **FundingSource** (довідники) — категорії витрат і джерела коштів.
- **Document** — файл (`upload`) або зовнішнє посилання (`link`), прив'язаний до авто та/або витрати.

Повна модель даних — у [database.md](database.md).

## Поза межами першої версії

Свідомо не реалізовано (із планом повернутися — деталі в [architecture-decisions.md](architecture-decisions.md)):

- окрема таблиця детального аудиту (`audit_log` з diff) — наразі аудит через `created_by`/`updated_by`/`deleted_by` + таймстемпи;
- відновлення пароля користувачем через e-mail — адмін скидає пароль вручну;
- інтеграція із зовнішнім API курсів валют — курси беруться з локального JSON;
- повноцінний CSRF-захист (token / double-submit) і денілист refresh-токенів для logout усіх пристроїв;
- фонове прибирання осиротілих файлів у сховищі після soft delete;
- структуровані JSON-логи й зовнішній моніторинг.
